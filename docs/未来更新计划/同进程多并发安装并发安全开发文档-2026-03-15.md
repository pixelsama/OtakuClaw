# 同进程多并发安装并发安全开发文档（2026-03-15）

## 1. 目标

在 **不依赖 UI 限制** 的前提下，让桌面端下载/安装链路具备如下能力：

- 同一应用进程内可并发执行多个安装任务。
- 并发执行时保持数据一致性、状态可预测、错误可隔离。
- 对现有 IPC 与设置页影响最小，优先主进程实现并发安全。

本方案聚焦“同进程并发安全”，不覆盖“重启后自动续传/恢复”。

## 2. 当前痛点（代码层）

1. 语音模型安装流程缺少全局并发调度器。  
   位置：`desktop/electron/services/voice/voiceModelLibrary.js`
2. `this.state` 与 `persistState()` 在并发任务下存在覆盖风险（最后写入者覆盖）。
3. `PythonEnvManager.ensureEnv(...)` 缺少按 `envId` 去重互斥，并发创建同 env 可能互踩。
4. 下载目标文件路径稳定化后（同 URL -> 同目标文件）缺少文件级互斥，可能出现 `.part` 冲突。
5. 安装完成后自动更新 selected bundle，多个任务并发完成时会出现“后完成覆盖先完成”。

## 3. 设计原则

- **任务并发，资源串行**：允许任务并行，但同一资源只允许一个写者。
- **单写入口**：所有可持久化状态变更经过统一状态仓（Store）。
- **幂等化**：任务重入、重复触发、网络抖动都不应破坏状态。
- **最小入侵**：先兼容现有 IPC，再逐步开放更强能力。

## 4. 目标架构

### 4.1 DownloadInstallTaskManager（新增）

在 Electron 主进程新增统一任务管理器（建议目录：`desktop/electron/services/download/`）：

- 注册任务：`createTask(kind, payload)`
- 状态机：`pending -> running -> completed|failed|canceled`
- 事件广播：统一输出 `download-task:progress`
- 并发控制：
  - 全局并发上限（例如默认 3）
  - 资源锁（见 4.2）

### 4.2 资源锁模型（新增）

引入逻辑锁（内存级，必要时扩展文件锁）：

- `env:<envId>`：Python env 创建/安装互斥
- `artifact:<absPath>`：目标下载文件互斥
- `bundle:<bundleId>`：bundle 目录写入互斥
- `voice-state`：voice state 持久化写入串行
- `nanobot-runtime`：保持现有单实例安装互斥

任务执行时按资源申请锁，释放遵循 finally 语义。

### 4.3 StateStore 单写通道（重构）

`VoiceModelLibrary` 的 `this.state` 改为由 `VoiceModelStateStore` 管理：

- `applyPatch(fn)`：串行执行 patch，内部保证写顺序
- `snapshot()`：只读快照
- `persist()`：内部节流 + 写队列

这样即便多个安装任务并发完成，也不会发生写覆盖。

### 4.4 选中模型策略调整

将“安装完成自动选中”改为可配置策略：

- 默认：`manual`（仅安装，不自动切换 selected）
- 可选：`auto-activate-if-idle`（无活跃选择变更时再切）

避免并发完成造成 selected 指针抖动。

## 5. IPC 与事件改造

### 5.1 新增统一事件通道（建议）

- `download-task:progress`
- payload 标准字段：
  - `taskId`
  - `taskType` (`voice-model-install` / `nanobot-runtime-install` / ...)
  - `phase`
  - `progress`
  - `currentFile`
  - `error`
  - `resourceLocks`

### 5.2 兼容现有事件（过渡期）

- 保留 `voice-models:download-progress`
- 保留 `nanobot-runtime:progress`
- 由统一任务管理器 fan-out 到旧通道，降低改造风险。

## 6. 实施拆分与工作量

## 阶段 A：并发基础设施（2-3 人天）

- 新建 `DownloadInstallTaskManager`
- 新建锁管理器 `ResourceLockManager`
- 新建基础任务状态机与事件发射

## 阶段 B：语音模型链路接入（3-4 人天）

- `VoiceModelLibrary` 接入任务管理器
- 下载、解压、写 bundle、写 state 分段加锁
- 去掉并发条件下不安全的直接写状态路径

## 阶段 C：Python env 并发安全（1-2 人天）

- `PythonEnvManager.ensureEnv` 增加按 `envId` 去重 promise map
- 重复任务共享同一构建流程结果

## 阶段 D：验证与回归（1 人天）

- 单测、集成测试、竞态压测
- 日志与错误码补齐

## 合计

- **约 7-10 人天**

## 7. 验收标准

1. 同时触发 3 个语音安装任务，全部完成后 state 文件无损坏、bundle 列表一致。
2. 同一 `envId` 并发 `ensureEnv`，只执行一次真实安装流程。 
3. 同一下载目标路径并发请求，不发生 `.part` 冲突或文件损坏。 
4. 任一任务失败不影响其他任务继续执行。
5. 旧 UI 仍可正常显示下载进度。

## 8. 测试策略

- 单元测试：
  - 锁申请/释放时序
  - 状态机迁移合法性
  - state store 串行写保证
- 集成测试：
  - 并发安装 + 并发删除
  - 失败注入（网络中断、解压失败、pip 失败）
- 压测：
  - 多任务高频触发下无崩溃、无 JSON 损坏

## 9. 风险与缓解

1. 风险：历史 IPC 强绑定旧事件格式。  
   缓解：保留旧事件，统一管理器只做上游。
2. 风险：自动选中策略变化引发体验差异。  
   缓解：默认 manual，并在设置层给出显式切换。
3. 风险：锁粒度过粗导致并发收益不明显。  
   缓解：先保守正确，后续按监控拆细锁粒度。

2. M2（第 2 周）：env 去重 + 全量回归 + 发布开关
