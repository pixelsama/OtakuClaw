# 进阶方案：可恢复/可取消/重启后继续开发文档（2026-03-15）

## 1. 目标

在“同进程并发安全”基础上，进一步实现：

- 任务可取消（用户主动中止）。
- 任务可恢复（进程内失败后重试/继续）。
- 应用重启后任务可继续（至少从可恢复阶段继续）。

该方案是上一份基础方案的增强层，不建议独立实施。

## 2. 能力定义

### 2.1 可取消

- 用户可对运行中任务发送 `cancel`。
- 任务进入 `canceling`，最终到 `canceled`。
- 需要保证：
  - 子进程被终止
  - 下载流被中断
  - 临时文件与锁被正确释放

### 2.2 可恢复

- 同进程内重试时复用已有阶段成果：
  - 已下载完整文件不重复下载
  - 已完成 env 安装直接复用
  - 已解压完成目录通过校验后跳过

### 2.3 重启后继续

- 任务元数据持久化到磁盘。
- 启动时扫描未终态任务并恢复执行。
- 从“最近安全检查点”继续，而不是盲目续流。

## 3. 关键设计

### 3.1 任务持久化日志（Task Journal）

建议新增目录：`<userData>/download-tasks/`

- `tasks.jsonl` 或 `tasks/<taskId>.json`
- 每个任务记录：
  - `taskId`
  - `taskType`
  - `createdAt`
  - `status`
  - `payload`
  - `checkpoint`
  - `attempt`
  - `lastError`

### 3.2 检查点（Checkpoint）模型

按安装流水线切分可恢复阶段：

1. `downloaded:<artifactKey>`
2. `extracted:<artifactKey>`
3. `env_ready:<envId>`
4. `bundle_written:<bundleId>`
5. `state_persisted`

恢复时按 checkpoint 向后执行，前置阶段只做校验不重做。

### 3.3 取消机制

为每个任务注入 `AbortController` + 子进程句柄注册：

- 下载器支持 `signal`
- Python/系统命令支持 kill tree
- 解压阶段可中断或阶段性原子提交

取消是“协作式 + 强制式”混合：

- 先请求协作中断
- 超时后强杀

### 3.4 重启恢复流程

应用启动时：

1. 读取任务 journal
2. 将 `running/canceling` 统一置为 `recovering`
3. 校验锁与文件现场
4. 生成恢复队列并调度执行
5. 广播恢复态进度事件

### 3.5 数据一致性策略

- 采用“临时目录 + 原子 rename”提交制
- `state.json` 采用“写临时文件后 replace”
- 每个 checkpoint 写入必须先 fsync（关键路径）

## 4. IPC 扩展建议

新增：

- `download-task:list`
- `download-task:cancel`
- `download-task:retry`
- `download-task:resume`
- `download-task:subscribe`

状态事件建议字段：

- `status`: `pending|running|canceling|canceled|recovering|completed|failed`
- `checkpoint`
- `resumable`: `true|false`
- `canCancel`: `true|false`

## 5. 与现有模块的改造点

1. `resumableDownloader`：增加 `AbortSignal`、恢复策略回调、校验钩子。
2. `VoiceModelLibrary`：安装阶段显式 checkpoint 化。
3. `PythonEnvManager`：env 创建支持恢复语义（存在即校验通过）。
4. `NanobotRuntimeManager`：下载/解压/pip 三段 checkpoint。
5. 新增 `TaskJournalStore` 与 `RecoveryCoordinator`。

## 6. 实施拆分与工作量

## 阶段 E：取消能力（2-3 人天）

- 下载与子进程中断通道
- 任务状态迁移与资源清理

## 阶段 F：检查点与恢复（3-4 人天）

- 任务 checkpoint 持久化
- 同进程重试/恢复逻辑

## 阶段 G：重启恢复（3-4 人天）

- 启动恢复协调器
- 任务重建与调度

## 阶段 H：稳定性与压测（2-3 人天）

- 故障注入
- 大文件中断恢复测试
- 崩溃恢复测试

## 合计（增强部分）

- **约 12-18 人天**（在基础并发安全方案之上）

> 若从零一次性实施“基础并发安全 + 进阶恢复能力”，总工作量通常会落在 18-26 人天。

## 7. 验收标准

1. 运行中任务可在 3 秒内进入 `canceling`，并在合理时间内进入 `canceled`。  
2. 下载中断后重试，已完整分片不重复下载。  
3. 应用强制退出后重启，未终态任务可恢复并继续推进。  
4. 恢复任务最终产物与全新安装产物一致（校验 hash/关键文件集）。

## 8. 测试矩阵

- 功能：取消、重试、恢复、重启继续
- 稳定性：网络波动、磁盘空间不足、权限变化
- 一致性：并发取消 + 并发安装 + 重启恢复组合场景
- 回归：旧事件通道兼容、UI 进度展示兼容

## 9. 风险与缓解

1. 风险：checkpoint 设计不当导致“假恢复成功”。  
   缓解：每个 checkpoint 必须绑定校验器（文件存在 + 大小/hash + 元数据）。
2. 风险：取消时资源泄漏（僵尸进程、锁未释放）。  
   缓解：统一清理栈 + finally 兜底 + 守护巡检。
3. 风险：重启恢复造成任务风暴。  
   缓解：恢复任务分批调度，限制启动并发。

## 10. 发布建议

1. 第 1 阶段：仅内部开关（`DOWNLOAD_TASK_RECOVERY_EXPERIMENT=1`）
2. 第 2 阶段：灰度用户启用恢复，不启用自动重试
3. 第 3 阶段：全量启用，保留降级开关
