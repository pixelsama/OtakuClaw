# Free-Agent-Vtuber-openclaw 桌宠功能技术指导（方案A，参考 Open-LLM-VTuber-Web）

## 1. 文档目标
本文件用于指导 `Free-Agent-Vtuber-openclaw` 在当前架构下实现桌宠模式（Pet Mode），并满足以下约束：
1. 以 `Open-LLM-VTuber-Web` 为“实现思路参考”，不直接复制其代码。
2. 采用方案A：单窗口（始终 `frameless + transparent`）+ 模式切换。
3. 允许未来演进到双窗口，但当前不引入双窗口复杂度。
4. 桌宠形态与主界面形态可以明显不同（独立 UI 壳层）。

## 3. 结论与总体方案
### 3.1 选型结论
采用方案A：单 `BrowserWindow`，固定无边框透明窗口，通过“窗口属性 + 前端壳层”切换模式。

### 3.2 为什么适合本项目
1. 当前聊天流、设置、Live2D 均在单前端状态树，单窗口改造成本最低。
2. 可避免双窗口带来的状态同步、WebGL 重建、事件路由复杂度。
3. 与“未来双窗口概率低，但 UI 差异大”的目标兼容：通过 `MainShell` / `PetShell` 实现形态分离。

### 3.3 架构原则
1. 主进程管窗口行为（置顶、穿透、任务栏、焦点、多屏 bounds）。
2. preload 仅暴露最小必要 API。
3. 渲染层通过 `mode` 选择壳层，不在一个页面堆大量样式分支。
4. 共享能力（聊天、设置、Live2D runtime）与 UI 壳层解耦。

---

## 4. 当前项目差距评估（基线）
当前已有：
1. Electron 主进程与 preload 基础链路（`desktop/electron/main.js`、`desktop/electron/preload.js`）。
2. 渲染进程桥接（`front_end/src/services/desktopBridge.js`）。
3. Live2D 组件基础能力（`front_end/src/components/live2d/Live2DViewer.jsx`）。

当前缺失：
1. 模式状态机（`window/pet`）与统一窗口管理模块。
2. 托盘模式切换菜单（Tray + Context Menu）。
3. 桌宠模式穿透逻辑（默认穿透 + 交互区域暂时取消穿透）。
4. 模式切换握手（pre-change / renderer-ready / changed / rendered）。
5. 主窗口 UI 与桌宠 UI 的壳层分离。

## 6. 主进程设计
## 6.1 `windowModeManager.js`
职责：
1. 创建窗口（固定 `frame: false`、`transparent: true`）。
2. 维护 `currentMode: 'window' | 'pet'`。
3. 维护 `windowedBounds`（用于从桌宠模式恢复）。
4. 管理鼠标穿透状态：
   - `hoveringComponents: Set<string>`
   - `forceIgnoreMouse: boolean`
5. 计算多屏联合区域并在 Pet 模式覆盖。

关键行为：
1. `setMode('window')`
   - `alwaysOnTop: false`
   - `skipTaskbar: false`
   - `focusable: true`
   - `resizable: true`
   - `ignoreMouseEvents: false`
2. `setMode('pet')`
   - 保存 `windowedBounds`
   - `alwaysOnTop: true`（建议层级 `screen-saver`）
   - `skipTaskbar: true`
   - `focusable: false`
   - `resizable: false`
   - `ignoreMouseEvents: true`（默认穿透）
   - 设置为多屏联合 bounds

## 6.2 `trayManager.js`
职责：
1. 创建托盘图标。
2. 菜单项包含：
   - Window Mode / Pet Mode（单选）
   - Toggle Mouse Passthrough（仅 Pet 模式展示）
   - Show / Hide / Exit
3. 模式切换通过回调调用 `windowModeManager.setMode(mode)`。

## 6.3 模式切换握手（防闪烁）
建议采用四步握手：
1. 主进程发 `pre-mode-changed`。
2. 渲染层完成一次 UI 切壳，回传 `renderer-ready-for-mode-change`。
3. 主进程应用窗口属性并发 `mode-changed`。
4. 渲染层完成最终渲染后回传 `mode-change-rendered`。

可选增强：
1. 切换期间窗口透明度降到 `0`，完成后恢复 `1`，减少视觉抖动。

---

## 7. preload 与 bridge 设计
扩展 `desktop/electron/preload.js` 暴露最小 API：
1. `windowMode.setMode(mode)`
2. `windowMode.getMode()`
3. `windowMode.onPreModeChanged(handler)`
4. `windowMode.onModeChanged(handler)`
5. `windowMode.notifyRendererReady(mode)`
6. `windowMode.notifyModeRendered()`
7. `windowMode.updateComponentHover(componentId, isHovering)`
8. `windowMode.toggleForceIgnoreMouse()`
9. `windowMode.onForceIgnoreMouseChanged(handler)`

同时在 `front_end/src/services/desktopBridge.js` 增加对应封装，保持 renderer 不直接依赖原始 `ipcRenderer`。

## 8. 前端实现策略（重点：双壳层）
## 8.1 Mode Context
新增 `ModeContext`：
1. 统一管理 `mode: 'window' | 'pet'`。
2. 对外只暴露 `mode` 与 `setMode`。
3. Electron 环境下通过 bridge 发起切换；Web fallback 固定 `window`。

## 8.2 壳层拆分
1. `MainShell`
   - 当前配置面板、文本输入、控制区等主应用 UI。
2. `PetShell`
   - 极简 UI（Live2D + 字幕 + 少量控制）。
   - 非必要面板默认不挂载，减少误触和视觉噪音。

当前单窗口实现方式：
1. `App.jsx` 内根据 `mode` 二选一挂载 `MainShell` / `PetShell`。
2. 共享服务（聊天流、设置）继续复用现有 hook。

## 8.3 鼠标穿透与交互区域
策略：Pet 模式默认穿透，仅在可交互区域取消穿透。

实现点：
1. 在 `PetShell` 中为交互组件（模型、输入框、按钮）绑定 hover 生命周期。
2. `mouseenter` -> `updateComponentHover(id, true)`
3. `mouseleave` -> `updateComponentHover(id, false)`
4. 主进程根据 `hoveringComponents.size` 动态设置 `setIgnoreMouseEvents(...)`。

## 8.4 拖拽
建议在 `PetShell` 的模型容器加拖拽 hook：
1. 按下时进入拖拽态。
2. 移动时更新元素 transform（仅渲染层位移）。
3. 拖拽结束时恢复 hover 状态同步。

备注：初版可只做“模型在窗口内拖拽”，不做位置持久化；后续再加。

---

## 9. IPC 协议草案（建议）
建议统一 `pet:*` 命名，便于未来拆窗口时保持兼容：

1. `pet:set-mode`（invoke）
- 入参：`{ mode: 'window' | 'pet' }`
- 出参：`{ ok: true, mode }`

2. `pet:get-mode`（invoke）
- 出参：`{ mode }`

3. `pet:pre-mode-changed`（event, main -> renderer）
- payload：`{ mode }`

4. `pet:renderer-ready`（event, renderer -> main）
- payload：`{ mode }`

5. `pet:mode-changed`（event, main -> renderer）
- payload：`{ mode }`

6. `pet:mode-rendered`（event, renderer -> main）
- payload：`{ mode }`

7. `pet:update-hover`（event, renderer -> main）
- payload：`{ componentId, isHovering }`

8. `pet:toggle-force-ignore-mouse`（event）

9. `pet:force-ignore-mouse-changed`（event, main -> renderer）
- payload：`{ forceIgnoreMouse }`

## 10. 分阶段实施计划
## Phase 0：重构准备（0.5 天）
1. 新建 `windowModeManager` 与 `trayManager` 文件骨架。
2. 将 `main.js` 中窗口创建逻辑迁移到 manager。
3. 保持原功能可运行，不引入行为变化。

## Phase 1：基础桌宠模式（1 天）
1. 窗口切到固定 `frameless + transparent`。
2. 打通 `window/pet` 切换 IPC。
3. 引入托盘菜单切换模式。
4. 完成主进程窗口属性切换。

## Phase 2：交互可用性（1 天）
1. 接入模式切换握手，降低闪烁。
2. 实现“默认穿透 + hover 取消穿透”。
3. 实现模型拖拽（PetShell）。

## Phase 3：形态分离（1 天）
1. 拆分 `MainShell` / `PetShell`。
2. 将现有主界面组件仅留在 `MainShell`。
3. `PetShell` 保留必要字幕/最小输入能力。

## Phase 4：稳定性与测试（1 天）
1. 增加主进程单元测试（模式切换状态、穿透状态机）。
2. 增加前端测试（mode context、桥接行为）。
3. 完成手工回归清单。

## 12. 风险与对策
1. 无边框后主窗口可用性下降  
- 对策：主模式提供自定义标题栏与窗口控制按钮。

2. 透明窗口在不同平台兼容性差异  
- 对策：先在目标平台（Windows/Linux）各做一次专项回归。

3. 穿透逻辑导致“点不到模型”或“无法退出”  
- 对策：托盘保留“Toggle Mouse Passthrough”和“Show”兜底项。

4. 模式切换造成 UI 闪烁或失焦  
- 对策：使用握手流程和透明度过渡；对焦点设置做平台分支处理。

## 13. 未来若要升级到双窗口，如何平滑迁移
在本方案中预留以下扩展点即可：
1. `windowModeManager` 内部接口保持语义化（`setMode`, `setIgnoreMouse`, `show/hide`）。
2. IPC 频道保持 `pet:*` 语义，不绑死“唯一主窗口”。
3. 前端保持 `MainShell` / `PetShell` 物理分离。

迁移步骤：
1. 新增 `PetWindowManager`，将 `PetShell` 挂载到第二窗口。
2. 现有 `setMode('pet')` 改为“隐藏 MainWindow + 显示 PetWindow”。
3. bridge 协议不变，只改路由目标窗口。
