# 自动更新实施计划（2026-03-14）

## 背景

当前仓库已通过 GitHub Actions 产出 macOS 安装包与更新元数据（`latest-mac.yml`、`*.blockmap`），但应用内尚未接入自动更新逻辑（未接入 `electron-updater`）。

## 目标

- 在不新增自建服务器的前提下，为 macOS 版本提供应用内自动更新能力。
- 优先使用 GitHub Releases 作为更新源。
- 保持用户可控：可检查更新、可手动触发下载、可确认后安装重启。

## 范围

- 平台范围：macOS（Apple Silicon）优先。
- 发布范围：仅正式 release 参与自动更新；`pre-release` 默认不推送给普通用户。
- 不在本期范围：Windows/Linux 自动更新接入。

## 实施方案

1. 主进程接入更新模块  
   - 引入 `electron-updater`。  
   - 实现 `checkForUpdates`、`downloadUpdate`、`quitAndInstall` 生命周期。  
   - 监听并转发状态事件：`checking-for-update`、`update-available`、`download-progress`、`update-downloaded`、`error`。

2. IPC 与前端交互  
   - 新增 IPC：  
     - `app-updater:check`  
     - `app-updater:download`  
     - `app-updater:install`  
     - `app-updater:state`（事件推送）  
   - 在设置页增加“检查更新”入口与下载进度展示。

3. 发布策略  
   - 使用 GitHub Releases 作为更新源。  
   - 工作流继续产出并上传：`dmg`、`latest-mac.yml`、`dmg.blockmap`。  
   - 可选保留：`checksums-macos.txt`（供手工校验）。

4. 版本与通道策略  
   - 语义化版本（如 `0.2.2-beta.1`、`0.2.2`）。  
   - 普通用户默认仅接收稳定通道（非 prerelease）。  
   - 后续可新增“测试通道”开关（显式允许 beta 更新）。

## 验收标准

- 应用能正确检测到比当前版本更新的 GitHub Release。
- 下载进度可见，下载完成后可一键安装重启。
- 对网络失败、权限失败、签名异常有明确报错文案。
- 从 `vX.Y.Z` 升级到 `vX.Y.Z+1` 的链路在实机可复现通过。

## 成本与基础设施

- 默认无需新增服务器成本：可直接使用 GitHub Releases 分发更新元数据与安装包。
- 仅在以下场景考虑自建分发：私有分发、超大规模下载、灰度策略更细化。

## 风险与回滚

- 风险：更新元数据与资产不一致导致更新失败。  
  - 规避：发布流程中增加资产完整性检查。
- 风险：误发有问题版本。  
  - 规避：先走 beta 通道验证，再推稳定通道。
- 回滚：发布紧急修复版本，或在客户端临时关闭自动检查入口。

## 里程碑

- M1：主进程自动更新能力接入 + IPC 打通  
- M2：前端更新 UI 与用户提示完成  
- M3：灰度发布验证（beta）  
- M4：稳定通道正式启用
