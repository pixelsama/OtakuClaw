### 4.1 假聊天后端

为了不依赖真实 OpenClaw/Nanobot，起了一个本地假 SSE 服务，监听：

`http://127.0.0.1:18789`

接口：

1. `POST /v1/chat/completions`
2. `GET /__stats`

行为：

1. 收到聊天请求后，返回两段 SSE：
   - `测试回复第一段。`
   - `测试回复第二段！`
2. `/__stats` 返回请求计数和最近一次请求体

当时使用的启动命令：

```sh
node -e "const http=require('http');let count=0;let lastBody='';const server=http.createServer((req,res)=>{if(req.url==='/__stats'){res.writeHead(200,{'content-type':'application/json'});res.end(JSON.stringify({count,lastBody}));return;}if(req.method==='POST'&&req.url==='/v1/chat/completions'){let body='';req.on('data',c=>body+=c);req.on('end',()=>{count+=1;lastBody=body;console.log('REQUEST',count,body);res.writeHead(200,{'content-type':'text/event-stream','cache-control':'no-cache','connection':'keep-alive'});const chunks=['测试回复第一段。','测试回复第二段！'];let i=0;const timer=setInterval(()=>{if(i<chunks.length){res.write('data: '+JSON.stringify({choices:[{delta:{content:chunks[i]}}]})+'\n\n');i+=1;return;}res.write('data: [DONE]\n\n');clearInterval(timer);res.end();},250);});return;}res.statusCode=404;res.end('not found');});server.listen(18789,'127.0.0.1',()=>console.log('FAKE_SERVER_READY'));"
```

## 7. 当前确认存在的两个真实问题

## 7.1 问题一：PTT 语音识别成功后，没有提交到聊天链路

这是当前最关键的 bug。

### 证据

1. GUI 实测中存在 `asr-final`
2. 但 fake backend `/__stats` 一直是：

```json
{"count":0,"lastBody":""}
```

3. 没有任何 `chat` 通道事件

### 最可疑代码位置

`front_end/src/hooks/voice/useVoiceMicToggle.js`

重点段：

1. `waitForSpeechQueue()` 附近
2. `stopPttCaptureAndSubmit()` 附近
3. `submitPttTranscription()` 附近

具体参考：

1. [front_end/src/hooks/voice/useVoiceMicToggle.js#L272](/Users/pixelsama/Dev/Github/Free-Agent-Vtuber-Openclaw/front_end/src/hooks/voice/useVoiceMicToggle.js#L272)
2. [front_end/src/hooks/voice/useVoiceMicToggle.js#L296](/Users/pixelsama/Dev/Github/Free-Agent-Vtuber-Openclaw/front_end/src/hooks/voice/useVoiceMicToggle.js#L296)
3. [front_end/src/hooks/voice/useVoiceMicToggle.js#L346](/Users/pixelsama/Dev/Github/Free-Agent-Vtuber-Openclaw/front_end/src/hooks/voice/useVoiceMicToggle.js#L346)

### 当前判断

高度怀疑是 flush / queue 时序问题：

1. `stopPttCaptureAndSubmit()` 先 `await stopVad()`
2. 然后只 `await Promise.race([speechQueueRef.current, timeout])`
3. 但最后一个 `onSpeechEnd` 很可能在 `stopVad()` 返回后才真正入队
4. 于是 `submitPttTranscription()` 看见的是空的 `segmentTextsRef`
5. 最终 ASR 明明成功了，但聊天没发出去

换句话说：

1. 真实 bug 不是“麦克风没采到”
2. 也不是“ASR 没跑”
3. 而是 “最后一段 ASR 结果没有被可靠收敛到 PTT 提交”

## 9. 下一轮建议直接做的修复

推荐先修顺序：

1. 修 PTT flush/queue 时序
2. 补前端测试覆盖这个竞态
3. 再修字幕 DOM 不更新
4. 然后复跑同样的 GUI 自动化

### 9.1 PTT 修复建议

目标：

1. 松手后必须保证尾段被收敛
2. `submitPttTranscription()` 执行前，所有可能的尾段入队都已经结束

建议方向：

1. `stopVad()` 后不要只等当前 `speechQueueRef.current`
2. 需要一个真正的 flush barrier，覆盖“stop 导致的最后一次 `onSpeechEnd`”
3. 可以考虑显式保存“in-flight speech task count”或“pending segment count”
4. 直到该计数稳定为 0，或 flush 超时，再提交
5. 然后把来源改成 `voice-ptt`

### 9.2 字幕修复建议

先在 `useStreamingSubtitleBridge` 上做更细的日志：

1. `onSegmentReady` 是否真正调用 `setSegmentText`
2. `onDelta` 是否被 `segmentModeRef.current` 吞掉
3. `onDone` 是否过早 `finishStream()` / `clearSubtitle()`
4. `voice segment tts lifecycle` 是否把 pending 状态推进错了
