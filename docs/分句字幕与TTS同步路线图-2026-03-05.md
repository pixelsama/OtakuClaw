## 3. 目标终态

实现“同一段文本只切一次，整链路复用”的统一模型：

1. Chat 生成 `segment`。
2. TTS 消费同一 `segment` 队列（不再二次切句）。
3. 字幕按同一 `segmentId` 显示。
4. 播放状态（started/finished/failed）与字幕可严格对齐。
5. 用户仅可见最终回复文本，不可见工具调用中间态。

## 4. 统一事件协议（冻结草案）

## 4.1 Segment 对象

```ts
type Segment = {
  sessionId: string
  turnId: string
  segmentId: string
  index: number
  text: string
  final: boolean
  source?: 'openclaw' | 'nanobot'
  lang?: 'zh' | 'en' | 'auto'
  createdAt?: number
}
```

## 4.2 Chat 事件（`chat:stream:event`）

1. `text-delta`（兼容保留）
2. `segment-ready`（新主路径）
3. `done`
4. `error`

说明：前端字幕主路径应逐步过渡到 `segment-ready`，`text-delta` 保留给旧后端与应急回退。

## 4.3 Voice/TTS 事件（`voice:event`）

新增事件类型（建议）：

1. `segment-tts-started`
2. `segment-tts-finished`
3. `segment-tts-failed`

payload 最小字段：

```json
{
  "sessionId": "...",
  "turnId": "...",
  "segmentId": "...",
  "index": 0,
  "text": "..."
}
```

失败事件补充：

```json
{
  "code": "voice_tts_failed",
  "message": "...",
  "retriable": true
}
```

## 5. 后续阶段总览

### Phase 2：TTS 接入统一 segmentId（核心）

目标：TTS 播放阶段与 `segment-ready` 对齐到同一 `segmentId`。

任务：

1. 在主进程新增“segment 播放队列”状态结构（按 `sessionId + turnId` 管理）。
2. 当聊天产生 `segment-ready` 时，将 segment 入队（仅文本，不含音频）。
3. TTS 从队列逐条消费，发出：
   - `segment-tts-started`
   - `segment-tts-finished`
   - `segment-tts-failed`
4. 失败时保证不中断整个 turn（可配置：继续下一句或停止）。
5. 统一 stop/abort 行为：中断时对未播句发可追踪状态。

建议改动文件：

1. `desktop/electron/ipc/voiceSession.js`
2. `desktop/electron/main.js`（必要时串接 chat->voice）
3. `front_end/src/hooks/voice/useVoiceSession.js`
4. `front_end/src/hooks/voice/useVoiceTtsPlayback.js`

验收标准：

1. 每个 `segment-ready` 最终都对应一个终态：`finished` 或 `failed`（或 abort 标记）。
2. 事件中 `segmentId` 不丢失、不重复、不跨 turn 串线。
3. `voice:tts:stop` 能在 1 个 chunk 周期内停止当前句并正确收口。

### Phase 3：字幕完全切到 segment 生命周期

目标：字幕从“文本流展示”升级为“播放态展示”。

任务：

1. 字幕默认以 `segment-tts-started` 作为显示触发点。
2. `segment-tts-finished` 决定切下一句或隐藏延时。
3. `segment-tts-failed` 时可选策略：
   - 显示该句并标记静音
   - 直接跳过并提示错误图标
4. 保留 `segment-ready` 与 `text-delta` 作为无 TTS 时降级通道。

建议改动文件：

1. `front_end/src/hooks/chat/useStreamingSubtitleBridge.js`
2. `front_end/src/hooks/useSubtitleFeed.js`
3. `front_end/src/shells/PetShell.jsx` / `SubtitleBar.css`（根据播放态微调停留时长）

验收标准：

1. 字幕切句与实际播放句一致。
2. 不出现“上一句未播完，字幕已跳下一句”的错位。
3. 关闭 TTS 时字幕仍正常（`segment-ready` 路径）。

### Phase 4：分句策略参数化与配置化

目标：分句行为可调、可灰度。

任务：

1. 将分句参数抽成配置：
   - `maxChars`
   - 硬切标点集
   - 软切标点集
2. 支持按语言切分策略（zh/en/auto）。
3. 支持运行时热更新（设置页或 env）。

建议改动文件：

1. `desktop/electron/services/chat/segmenter.js`
2. `desktop/electron/services/settingsStore.js`
3. `front_end/src/components/config/ConfigDrawer.jsx`（如需 UI）

验收标准：

1. 同一输入在固定配置下分句结果稳定。
2. 极长句不会撑爆 UI（保证分句上限生效）。

### Phase 5：可观测性与问题诊断

目标：出现“字幕不同步/漏播”时可快速定位。

任务：

1. 为 segment 全链路打点：
   - ready 时间
   - tts-started 时间
   - tts-finished 时间
2. 增加 debug 开关（默认关闭）输出关联 ID。
3. 新增诊断面板（可后置）查看最近 20 条 segment 状态。

建议改动文件：

1. `desktop/electron/ipc/*`
2. `front_end/src/hooks/*`
3. `docs/` 增补 runbook

验收标准：

1. 任意 segment 可通过 ID 追踪完整生命周期。
2. 可在 5 分钟内定位“卡在 ready / started / finished 哪一段”。

### Phase 6：安全与输出治理收口

目标：确保不会再次把工具调用、思维链或内部路径暴露给用户。

任务：

1. 维持“只下发最终回复文本”策略（Nanobot bridge）。
2. 增加防回归测试：包含 `Tool call` 场景输入。
3. 增加输出净化规则（如未来恢复流式中间态时）。

建议改动文件：

1. `desktop/electron/services/chat/nanobot/nanobot_bridge.py`
2. `desktop/electron/tests/nanobotBackend.test.js`（新增泄露测试）

验收标准：

1. 用户 UI 不出现 `read_file/write_file/list_dir/edit_file` 等工具痕迹。
2. 终端日志可见与否由 debug 开关控制，不影响 UI 输出安全。

1. 先做 Phase 2（TTS 事件对齐），这是主价值。
2. 再做 Phase 3（字幕按播放态），解决“看见即播到”。
3. 接着做 Phase 6（安全回归），避免回退泄露问题。
4. 最后做 Phase 4/5（参数化与可观测性），提升维护效率。
