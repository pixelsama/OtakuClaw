# 内置 Python 语音栈迁移说明（2026-03-04）

## 目标

将桌面端语音能力从 `sherpa-onnx-node` 逐步迁移到内置 Python 生态，做到：

1. 用户无需手动安装 Python。
2. 应用内可下载并配置专用 Python 运行时。
3. 应用内可下载并绑定 ASR/TTS 模型。

## 本次已落地内容

### 1. 内置模型清单新增 Python 语音栈条目

`desktop/electron/services/voice/voiceModelCatalog.js`

新增 `builtin-python-funasr-qwen3tts-v1`：

- Python runtime: python-build-standalone 3.11.15（按平台下载）
- ASR 模型：`FunAudioLLM/SenseVoiceSmall`
- TTS 模型：`Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice`
- TTS tokenizer：`Qwen/Qwen3-TTS-Tokenizer-12Hz`

### 2. 模型库支持“运行时安装 + 模型拉取”

`desktop/electron/services/voice/voiceModelLibrary.js`

- `installCatalogBundle()` 在检测到 `runtime.kind=python` 时走 Python 安装流程。
- 下载并解压专用 Python 到 bundle 目录。
- 自动安装 Python 依赖（`funasr` / `qwen-tts` / `huggingface_hub` / `modelscope` 等）。
- 自动执行 bootstrap 脚本下载 ASR/TTS 模型。
- 把 Python 运行时、脚本路径、模型目录写入 bundle state。

### 3. providerFactory 新增 python provider

`desktop/electron/services/voice/providerFactory.js`

- `VOICE_ASR_PROVIDER=python` 时启用 Python ASR provider。
- `VOICE_TTS_PROVIDER=python` 时启用 Python TTS provider。

### 4. 新增 Python Bridge

- `desktop/electron/services/voice/providers/python/voice_bridge.py`
  - ASR：调用 FunASR（默认走 SenseVoice）
  - TTS：调用 Qwen3-TTS
- `desktop/electron/services/voice/providers/python/bootstrap_runtime.py`
  - 下载模型（优先 Hugging Face，失败可回退 ModelScope）

### 5. 新增 Node provider 实现

- `desktop/electron/services/voice/providers/asr/pythonProvider.js`
- `desktop/electron/services/voice/providers/tts/pythonProvider.js`

通过子进程调用 Python bridge，统一转换为现有 `voice:event` 协议。

## 使用方式

1. 打开桌面端语音设置。
2. 在“内置模型”中选择 `Python 语音栈（SenseVoice + Qwen3-TTS）`。
3. 点击安装并等待下载、依赖安装、模型拉取完成。
4. 选择该 bundle 后，语音链路将自动切换到 `python` provider。

## 当前限制

1. Qwen3-TTS 1.7B 资源开销较大，CPU 可运行但速度可能较慢。
2. 首次安装时间较长（Python 运行时 + pip 依赖 + 模型下载）。
3. 网络环境受限时，建议优先使用 ModelScope 源。
4. Qwen3-TTS 官方示例目前主要以 CUDA 环境为主，Mac 侧建议走 MPS/CPU 回退策略。
5. 已安装的旧 `Fun-ASR-Nano-2512` bundle 建议重新安装为新版 `SenseVoiceSmall` bundle。

## Mac 优先策略（本次升级）

1. Python bridge 设备自动选择策略升级为：`cuda:0 -> mps -> cpu`。
2. 在 Apple Silicon 上会优先尝试 `mps`，若模型或算子不兼容会自动回退 `cpu`。
3. ASR/TTS 均采用同一策略，确保“优先加速、失败可用”。
