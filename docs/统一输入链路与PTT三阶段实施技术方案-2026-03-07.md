# 统一输入链路与 PTT 三阶段实施技术方案（2026-03-07）

## 1. 目标与范围

本文将《统一输入链路与 PTT 语音交互重构报告（2026-03-07）》落地为三阶段实施方案，覆盖：

1. Phase 0：快速止血，修复字幕与语音链路不一致
2. Phase 1：引入 PTT（按住说话、松手一次提交）交互
3. Phase 2：输入/输出事件总线收敛与 session 内部化

本文只定义技术实施路径，不包含代码提交。

---

## 2. 当前基线（实施前）

### 2.1 已有能力

1. 文本与语音在进入 Nanobot 前均可走 `chat:stream:start`
2. 语音链路包含 VAD 分段、ASR 转写、TTS 播放与 ACK 背压
3. 模型选择与预热机制已存在（主进程支持 warmup）

### 2.2 当前主要问题

1. 文字输入与语音输入在“输出消费路径”上行为不一致
2. 语音自动流存在字幕监听时序风险
3. 自然停顿可触发多次 ASR commit，造成“用户一次意图 -> 多次请求”
4. `voice session` 暴露到 UI 主流程，状态耦合重、扩展成本高

---

## 3. 实施总原则

1. 小步上线：先稳定、再升级交互、最后做结构收敛
2. 合同优先：IPC 事件协议必须前向兼容，避免一次性破坏旧 UI
3. 可观测优先：每阶段补齐日志与测试，再进入下一阶段
4. 默认策略清晰：并发策略默认 `latest-wins`

---

## 4. Phase 0（快速止血）

### 4.1 目标

1. 保证语音自动触发流也能稳定显示字幕
2. 消除由监听懒初始化导致的“有 TTS 无字幕”异常

### 4.2 主要改动点

1. 将聊天事件订阅改为应用启动常驻订阅，避免依赖某条输入路径触发初始化
2. 保留现有 `voice:event` + `chat:stream:event` 双事件面，不做协议破坏
3. 修正前端桥接时序：确保 `segment-ready`、`text-delta`、`done/error` 全部可被消费

### 4.3 影响模块

1. `front_end/src/hooks/useStreamingChat.js`
2. `front_end/src/hooks/chat/useStreamingSubtitleBridge.js`
3. `front_end/src/App.jsx`

### 4.4 测试与验收

1. 回归语音自动流：出现 TTS 时字幕必须同步出现
2. 回归文本输入：字幕流保持原行为
3. 保持 `front_end/tests/streamingChatAdopt.test.js` 通过

### 4.5 预计规模

1. 工期：0.5-1.5 天
2. 风险：低

---

## 5. Phase 1（PTT 交互升级）

### 5.1 目标

1. 引入“按住说话、松手提交一次”的语音输入窗口
2. VAD 仅用于分段与静音过滤，不再决定提交边界
3. 降低自然停顿导致的多请求误触发

### 5.2 交互定义

1. `PTT_KEY_DOWN`：打开语音窗口，开始采集
2. 按住期间：VAD 分段，ASR 可并发处理各段
3. `PTT_KEY_UP`：停止采集并触发 flush
4. flush 完成：按段序拼接文本，仅发送一次 `chat:stream:start`

### 5.3 技术改动点

1. 新增 `VoiceInputWindow` 聚合状态（窗口 epoch、segment 队列、在途 ASR、flush 状态）
2. 增加段级保序拼接，显式使用 `segmentId` 排序
3. 增加 flush 总超时与超时降级策略（可提交已完成片段并提示）
4. 新增可配置 PTT 键位
5. 不将 `Fn` 作为唯一触发键（跨平台可捕获性不稳定）

### 5.4 IPC/协议建议

1. 保持既有 `voice:*` IPC 可用
2. 新增或扩展 PTT 控制 IPC（开始窗口/结束窗口/flush）
3. 不变更 `chat:stream:start` 入参语义，继续传 `content` 文本

### 5.5 影响模块

1. `front_end/src/hooks/voice/useVoiceMicToggle.js`
2. `front_end/src/hooks/voice/useSileroVad.js`
3. `desktop/electron/ipc/voiceSession.js`
4. `desktop/electron/preload.js`
5. `front_end/src/services/desktopBridge.js`

### 5.6 测试与验收

1. 一次按住松手，最多触发一次聊天请求
2. 中途自然停顿不触发额外请求
3. 松手后 flush 在超时内完成（成功或可解释失败）
4. 语音链路与 TTS 背压 ACK 机制无回归

### 5.7 预计规模

1. 工期：3-6 天
2. 风险：中

---

## 6. Phase 2（架构收口）

### 6.1 目标

1. 输入统一：文本与语音统一进入 `submitUserText(...)`
2. 输出统一：统一为 `conversation:event` 总线
3. session 内部化：UI 不再依赖复杂 `voice session` 生命周期语义

### 6.2 技术改动点

1. 主进程引入 `ConversationRuntime`（单实例）
2. 将 `chat:stream:event` 与语音相关输出映射到统一事件模型
3. 将字幕消费者与 TTS 消费者改为订阅同一输出总线
4. 明确并发策略：
   - 默认 `latest-wins`
   - 可选 `queue`，仅用于特定工作流

### 6.3 协议迁移策略

1. 先新增 `conversation:event`，并保留旧事件一段时间
2. 前端完成迁移后，再逐步下线旧监听路径
3. 迁移期间增加事件镜像日志，核对两条链路一致性

### 6.4 影响模块

1. `desktop/electron/main.js`
2. `desktop/electron/ipc/chatStream.js`
3. `desktop/electron/ipc/voiceSession.js`
4. `front_end/src/hooks/useStreamingChat.js`
5. `front_end/src/hooks/chat/useStreamingSubtitleBridge.js`
6. `front_end/src/App.jsx`

### 6.5 测试与验收

1. 文本输入与语音输入在字幕/TTS上的行为一致
2. 并发冲突下行为与策略一致（latest-wins 或 queue）
3. 后台主动消息可直接进入统一输出链路
4. 旧协议兼容窗口内无功能回退

### 6.6 预计规模

1. 工期：7-12 天
2. 风险：中高

---

4. 统一事件总线下字幕/TTS 同步
5. latest-wins 并发中断语义

## 8. 发布与回滚策略

1. 每个 Phase 独立发布，不跨阶段打包
2. 每阶段都保留开关或兼容路径，便于快速回滚
3. Phase 2 采用“双写/镜像”迁移，先观察再切主路

---

## 9. 里程碑建议

1. M1（Phase 0 完成）：链路稳定，异常可复现可消除
2. M2（Phase 1 完成）：PTT 可用，误触发显著下降
3. M3（Phase 2 完成）：统一事件架构上线，旧链路下线

---

## 10. 结论

推荐执行顺序为：`Phase 0 -> Phase 1 -> Phase 2`。  
该顺序可在控制风险的前提下，先解决用户可见问题，再升级交互，最后完成架构收敛。
