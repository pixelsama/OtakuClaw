# 桌宠截图权限后鼠标抓手粘滞（已知问题，待修复）

记录日期：2026-03-15  
状态：`Open`（仅记录，不在本轮修复）

## 问题概述

在 macOS 上，首次触发截图权限弹窗并授权后，应用重启；随后在桌宠模式执行一次截图并删除截图草稿后，切到其他应用会出现鼠标一直保持桌宠拖拽抓手样式、且点击无法作用于其他应用的问题。  
该问题不是本轮新增功能引入，为历史已存在缺陷。

## 复现步骤（用户实测）

1. 打开应用。
2. 点击截图按钮，系统弹出屏幕录制权限请求。
3. 在系统弹窗中允许权限，应用被系统退出并重开。
4. 重开后切换到桌宠模式。
5. 点击截图，再把截图删除（移除截图草稿）。
6. 打开另一个应用并尝试点击。

## 实际结果

- 鼠标持续显示为桌宠拖拽抓手样式。
- 其他应用点击无效，输入焦点/交互被桌宠“抢占”。
- 此时仍可操作桌宠（拖拽与其它按钮）。

## 期望结果

- 切换到其他应用后，鼠标样式恢复正常。
- 桌宠不应继续拦截其他应用点击。

## 临时绕过

在当前应用中执行一次：

1. 切回窗口模式；
2. 再切回桌宠模式。

执行后问题可恢复。

## 影响评估

- 严重级别建议：`P2`（高可见、影响多应用交互，但有明确临时绕过）。
- 影响场景：首次截图权限授权后的桌宠截图链路，且包含“截图后删除草稿”的操作序列。

## 初步排查方向（后续修复参考）

1. 核查截图权限授权重启后，`WindowModeManager` 的鼠标穿透状态是否与渲染层 hover 状态一致。  
2. 核查截图完成/删除流程中，桌宠相关 hover 标记（如 `pet-dragging`、`live2d-hitbox`、`pet-capture-preview`）是否存在未释放路径。  
3. 核查截图 overlay 生命周期结束后，是否遗漏了一次 `setIgnoreMouseEvents(false/true)` 的纠偏同步。  
4. 增加回归用例：覆盖“权限授权重启 -> 桌宠截图 -> 删除草稿 -> 切换外部应用交互”的状态机路径。
